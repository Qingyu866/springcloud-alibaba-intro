### 2.3 P0-3：多级缓存架构页面

**页面名称**：扩展 `RedisPage.tsx`（原898行 → 扩展到1200-1500行）
**预计工作量**：3-4 天

#### 新增内容大纲

**第一部分：缓存模式深入（新增200-250行）**
- Cache Aside 模式（旁路缓存）
  - 读流程：先查缓存，miss 则查 DB 并回写缓存
  - 写流程：先更新 DB，再删除缓存
  - 适用场景：读多写少
  - 代码实现
- Read Through 模式（读穿透）
  - 缓存层自己加载 DB
  - 应用层只与缓存交互
  - 适用场景：读多写少，简化应用层
- Write Through 模式（写穿透）
  - 写缓存时同步写 DB
  - 强一致性，性能较差
  - 适用场景：数据一致性要求高
- Write Behind 模式（异步写回）
  - 先写缓存，异步批量写 DB
  - 高性能，存在丢数据风险
  - 适用场景：写多读少，允许短时不一致

**第二部分：多级缓存架构（新增250-300行）**
- 为什么需要多级缓存
  - L1 本地缓存：极快访问（μs级）
  - L2 分布式缓存：快速访问（ms级）
  - L3 数据库：慢速访问（10-100ms级）
- 本地缓存选型
  - Caffeine（推荐，高性能）
  - Guava Cache
  - Ehcache（老牌）
- 多级缓存实现
  - L1 (Caffeine) + L2 (Redis)
  - 缓存穿透策略
  - 缓存同步策略
- 代码实现
  - Spring @Cacheable 配置
  - 自定义多级缓存管理器
  - 缓存预热

**第三部分：缓存一致性保证（新增200-250行）**
- 缓存更新策略对比
  - 先更新 DB，再更新缓存（❌ 线程不安全）
  - 先删除缓存，再更新 DB（❌ 并发读脏数据）
  - 先更新 DB，再删除缓存（✅ 推荐，仍需延时双删）
  - 延时双删（✅ 解决并发问题）
- 分布式环境下的一致性挑战
  - 主从延迟导致缓存不一致
  - 解决方案：Canal 订阅 binlog
- Canal 实战
  - Canal 架构
  - MySQL binlog 订阅
  - 缓存更新代码实现
- 最终一致性方案

**第四部分：缓存穿透、击穿、雪崩（新增150-200行）**
- 缓存穿透
  - 问题：查询不存在的 key，请求直达 DB
  - 方案1：布隆过滤器（Bloom Filter）
  - 方案2：缓存空对象（设置短过期时间）
- 缓存击穿
  - 问题：热点 key 过期，大量请求直达 DB
  - 方案1：互斥锁（Redis SETNX）
  - 方案2：热点 key 永不过期（逻辑过期）
- 缓存雪崩
  - 问题：大量 key 同时过期，DB 压力激增
  - 方案1：过期时间加随机值
  - 方案2：熔断降级
  - 方案3：高可用缓存架构

**第五部分：热点数据处理（新增150-200行）**
- 热点 key 发现
  - Redis Keyfreq 统计
  - 应用层统计
- 热点 key 处理
  - 本地缓存 + 分布式锁
  - 备份副本机制
- 热点更新实时推送
  - Redis Pub/Sub
  - Canal 实时推送
